### MSPT-11 — Cache headers for hashed assets and pipeline data

**Context.** The mobile audit (MA-08) measured production and found two Cloudflare
Pages defaults costing every visitor a round trip:

- `/data/hexbin.geojson` — the file that paints the map (1.05 MB, 118 KB brotli)
  — served `Cache-Control: public, max-age=0, must-revalidate`, so it revalidated
  on **every single visit**.
- `/assets/main-*.js` — content-hashed and immutable by construction — served
  `max-age=14400, must-revalidate`, expiring after four hours.

There was no `app/public/_headers` file to override either (only `_redirects`).

**Decisions.**

1. **Add `app/public/_headers`, not a Function.** Vite copies `public/` verbatim
   into `dist/`, exactly as it already does for `_redirects`. Two rules keep us
   far inside the 100-rule / 2,000-char-per-line Pages limits.

2. **`/assets/* → public, max-age=31536000, immutable`.** Vite fingerprints
   every file under `assets/` with a content hash, so changed content always
   means a changed URL. `immutable` additionally suppresses the revalidation
   request browsers otherwise send on a manual reload.

3. **`/data/* → public, max-age=86400, s-maxage=604800`.** The pre-aggregated
   pipeline outputs (`hexbin.geojson`, `trees.sample.geojson`,
   `species-counts.json`) are **not** content-hashed — a re-extract rewrites them
   at the same URL — so they cannot be `immutable`. Rather than invent a policy,
   this mirrors the one `functions/data/trees.geojson.js:27` already chose for
   the 51 MB point set: *a day at the browser, a week at the edge.*

4. **`_headers` cannot touch the R2 Function, and we rely on that.** Cloudflare
   documents that custom headers in `_headers` "are not applied to responses
   generated by Pages Functions, even if the request URL matches a rule." So the
   `/data/*` rule above never reaches `/data/trees.geojson`, which continues to
   set its own headers in `functions/data/trees.geojson.js`. That file remains the
   only place to retune the big point set. This is why a broad `/data/*` glob is
   safe here rather than an enumeration of the three static filenames.

5. **HTML is deliberately left alone.** Neither rule matches `/` or `/species`,
   so both keep the Pages default (`max-age=0, must-revalidate`). That is what
   lets a new deploy's hashed asset names be picked up on the very next visit —
   caching the HTML would strand clients on old bundles.

**Verification.** Built `app/dist`, then served it under `wrangler pages dev`
(Node, never Bun — see MSPT-2) and, as a control, an identical copy of the same
build with `_headers` deleted, reproducing today's production behaviour.

Wrangler reported `Parsed 2 valid header rules`. Response headers landed on the
intended path classes only — hashed assets `max-age=31536000, immutable`; the
three data files `max-age=86400, s-maxage=604800`; `/` and `/species` untouched at
`max-age=0, must-revalidate`; `/hexbin` still 308s.

The acceptance criterion ("a second visit re-uses `/assets/*.js` and
`/data/hexbin.geojson` from cache with no revalidation round trip") was then
checked by driving Chromium with a persistent on-disk cache, loading the page
twice, and classifying each request via CDP as served-from-cache, disk-cache, or a
network round trip:

```text
control (no _headers)   second visit: both assets → network round trip   NOT met
treated (with _headers) second visit: both assets → disk-cache, no request   MET
```

**Tradeoff, stated plainly.** After a data re-extract, browsers may serve a stale
`hexbin.geojson` for up to 24 hours. That is the same exposure the 51 MB
`trees.geojson` has already accepted, and it is the direct cost of removing the
per-visit revalidation. Eliminating it properly means content-hashing the data
filenames (as Vite does for assets), which changes the fetch paths in
`app/src/hexbin.ts` and the `STATIC_DATA` staging step in
`scripts/deploy-cloudflare.mjs` — deliberately out of scope for this action.
